关键字

  1. void：用于声明一个方法没有返回值。

  2. private：访问修饰符，用于声明类的成员（变量或方法）只能在其所在的类内部访问。

  3. public：访问修饰符，用于声明类的成员可以被任何其他类访问。

  4. protected：访问修饰符，用于声明类的成员可以被同一个包中的类以及所有子类访问。

  5. static：用于声明类级别的变量或方法，它们不依赖于类的实例。

  6. final：用于声明一个变量的值不可变，或者一个方法不能被子类覆盖，或者一个类不能被继承。

  7. abstract：用于声明一个类或方法不能被实例化，或者一个方法没有具体的实现。

  8. interface：用于声明一个接口，它是一种形式的类，可以包含常量和抽象方法。

  9. implements：用于实现一个或多个接口。

  10. extends：用于声明一个类继承另一个类。

  11. try：用于捕获异常。

  12. catch：用于捕获try块中抛出的异常。

  13. finally：用于声明一段代码无论是否发生异常都会执行。

  14. throw：用于抛出一个异常。

  15. throws：用于声明一个方法可能抛出的异常。

  16. instanceof：用于检查一个对象是否是特定类的实例。

JAVA封装

在面向对象程式设计方法中，封装（英语：Encapsulation）是指一种将抽象性函式接口的实现细节部分包装、隐藏起来的方法。

封装可以被认为是一个保护屏障，防止该类的代码和数据被外部类定义的代码随机访问。

要访问该类的代码和数据，必须通过严格的接口控制。

封装最主要的功能在于我们能修改自己的实现代码，而不用修改那些调用我们代码的程序片段。

适当的封装可以让程式码更容易理解与维护，也加强了程式码的安全性。

JAVA接口

接口（英文：Interface），在JAVA编程语言中是一个抽象类型，是抽象方法的集合，接口通常以interface来声明。一个类通过继承接口的方式，从而来继承接口的抽象方法。

接口并不是类，编写接口的方式和类很相似，但是它们属于不同的概念。类描述对象的属性和方法。接口则包含类要实现的方法。

除非实现接口的类是抽象类，否则该类要定义接口中的所有方法。

接口无法被实例化，但是可以被实现。一个实现接口的类，必须实现接口内所描述的所有方法，否则就必须声明为抽象类。另外，在 Java 中，接口类型可用来声明一个变量，他们可以成为一个空指针，或是被绑定在一个以此接口实现的对象。

接口的继承

一个接口能继承另一个接口，和类之间的继承方式比较相似。接口的继承使用extends关键字，子接口继承父接口的方法。

接口多继承

在Java中，类的多继承是不合法，但接口允许多继承。

在接口的多继承中extends关键字只需要使用一次，在其后跟着继承接口。

泛型

场景：水果店的 “水果包装管理系统” 升级记

假设你是一家水果店的技术负责人，需要开发一套 “水果包装管理系统”—— 核心需求是用 “盒子” 装不同种类的水果（比如苹果、橘子），后续店员能从盒子里准确取出对应水果，避免拿错或损坏。下面结合你的代码，还原系统从 “原始版本” 到 “优化版本” 的升级过程：

一、旧系统：用 “万能盒子” 装水果，频繁踩坑

最开始，你为系统设计了一种 “万能盒子”（对应代码里的objectBox类）—— 这种盒子不挑水果类型，能装苹果、橘子，甚至装其他杂物。某天店员小王用这套系统处理苹果包装，流程是这样的：

  1. 准备工具：小王先拿了一个 “万能盒子”（new objectBox()），又从货架上取了一个新鲜苹果（new Apple()）；
  2. 装水果：他把苹果放进万能盒子里（ob.setContent(apple)），盒子上没贴任何 “装了什么” 的标签；
  3. 取水果：下午需要把苹果拿出来打包时，小王打开盒子，发现盒子里的东西是 “无类型标识” 的（对应Object类型）—— 他知道自己早上装的是苹果，所以只能 “赌一把”，强行认定里面是苹果（(Apple) ob.getContent()），然后拿出来用。

但这套系统很快出了问题：有次另一个店员小李误把橘子放进了 “万能盒子”，小王取的时候依然强行转成苹果，结果程序直接报错（ClassCastException）—— 就像 “把橘子硬当苹果切，不仅切坏水果，还导致后续订单无法完成”。原来 “万能盒子” 的问题在于：装的时候不限制类型，取的时候必须靠人 “记”，一旦记混或装错，就会出故障（对应代码里oldMethod的强转风险）。

二、新系统：用 “专属盒子” 装水果，安全又高效

为了解决旧系统的坑，你决定升级系统 —— 设计 “专属盒子”（对应代码里的newBox<T>泛型类）：每个盒子出厂时就明确 “只能装某一种水果”，比如 “苹果专属盒” 只能装苹果，“橘子专属盒” 只能装橘子，从根源上避免装错、取错。

升级后，店员小王处理苹果的流程变成了这样：

  1. 选对盒子：小王先拿了一个 “苹果专属盒”（newBox<Apple> AppleBox = new newBox<>()）—— 盒子上清晰印着 “仅装苹果” 的标识，系统也会在编译时检查 “盒子类型是否匹配水果”；
  2. 装水果：他把苹果放进专属盒（AppleBox.setContent(apple)）—— 这一步如果想放橘子，系统会直接报错（比如 “橘子不能放进苹果专属盒”），根本装不进去；
  3. 取水果：下午取苹果时，小王打开 “苹果专属盒”，直接就能拿出苹果（Apple content = AppleBox.getContent()）—— 不需要再 “赌” 或强行转换，因为盒子早就明确了 “里面一定是苹果”，拿出来就能直接用。

后续店里新增橘子品类时，流程也一样：店员只需拿 “橘子专属盒”（newBox<Orange> OrangeBox = new newBox<>()），装橘子、取橘子都不会和苹果混淆。这套新系统的核心优势是：装的时候 “锁死类型”，取的时候 “不用强转”—— 系统在编译阶段就帮人 “把关”，再也不会出现 “橘子当苹果” 的错误（对应代码里newMethod的泛型安全特性）。

三、系统升级的核心价值

从 “万能盒子” 到 “专属盒子” 的升级，本质是用泛型（newBox<T>）解决了 “类型安全” 和 “代码通用” 的矛盾：

  • 对店员来说：不用再靠 “记” 来区分盒子内容，操作更简单，出错率骤降；
  • 对系统来说：一套newBox类能适配所有水果（苹果、橘子、香蕉等），不用为每种水果单独写一个 “盒子类”，代码更简洁通用；
  • 对业务来说：后续新增水果品类时，不用修改盒子的核心逻辑，直接用 “新水果 + 专属盒” 即可，扩展性极强。

总结

使用反射而不是直接new一个对象的主要原因是反射提供了更大的灵活性和动态性，允许程序在运行时根据需要加载、操作类和对象。这在需要处理动态类名、操作私有成员、实现通用框架、调用动态方法以及避免代码重复等场景中非常有用。然而，反射也有一些缺点，如性能开销、破坏封装性和复杂性增加，因此在实际开发中应谨慎使用，权衡利弊。
